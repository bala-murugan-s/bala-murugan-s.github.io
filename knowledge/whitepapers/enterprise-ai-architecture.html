01
🧠 Scenario Description
Business Context

GlobalFinServ Corp

Industry: Financial Services (Banking & Trading Platform)

Scale: 8,500 employees · 42 branch offices across 12 countries

Users: Traders, analysts, back-office, remote workers, third-party partners

Cloud: AWS primary (us-east-1, eu-west-1) · Azure DR (secondary)

Compliance: PCI-DSS, SOX, GDPR, ISO 27001

Problem Statement

Pain Points Before This Architecture

  • Branch-to-cloud latency averaging 320ms for trading app
  • Manual firewall rule changes taking 3–5 days via change tickets
  • No unified visibility — 14 disparate log sources, no correlation
  • Mean Time to Detect (MTTD) incidents: 72 hours
  • No automated remediation — all responses manual and inconsistent
  • Branch WAN failures causing full outages (no failover)
High-Level Solution Overview

Deploy a unified, policy-driven architecture where:

  • SD-WAN provides intelligent path selection across MPLS + DIA + LTE at every branch, with sub-second failover and SLA-based routing per application class
  • NGFW (Palo Alto Prisma Access or Fortinet FortiGate) enforces zero-trust segmentation at every hop — branch, internet edge, and cloud perimeter
  • Kubernetes (EKS) hosts the trading platform, risk engine, and internal APIs with strict network policies, OPA admission control, and Falco runtime security
  • Terraform + Ansible codify all infrastructure and configuration as IaC, deployed via GitOps pipelines (GitLab CI)
  • Splunk Enterprise Security ingests logs from every layer — 40GB+/day — with real-time correlation rules and threat intelligence feeds
  • AI Agent (Claude-powered) continuously analyzes Splunk alerts, cross-correlates network/security/app telemetry, and triggers Ansible runbooks or Terraform changes autonomously

Layer 1 — Underlay

Physical connectivity per branch: MPLS (guaranteed SLA), DIA broadband (cost-optimized), LTE (emergency failover). BFD probes every 300ms detect link degradation before BGP converges.

Layer 2 — SD-WAN Overlay

IPsec full-mesh tunnels between branch vCPEs and regional hub. App-aware routing steers trading traffic to MPLS, web browsing to DIA, voice to QoS-prioritized path. SLA ≤ 10ms jitter, ≤ 0.1% loss.

Layer 3 — Security

NGFW enforces micro-segmentation: branches land in branch-untrust zone, traverse inspection, reach cloud-app zone only on approved policy matches. All TLS 1.3 traffic inspected.

Layer 4 — Cloud VPC

Hub-and-spoke VPC design via Transit Gateway. Route tables enforce that data-subnet (10.0.20.0/24) is only reachable from app-subnet. No public IPs on workload nodes. Egress via NAT Gateway.

Layer 5 — Kubernetes

EKS worker nodes in private subnets across 3 AZs. Istio sidecars enforce mTLS for all service-to-service. Falco detects container escapes in real time. OPA blocks privileged pod creation.

Layer 6 — AI + Automation

Splunk triggers webhook → Kafka topic → AI Agent. Agent queries context, constructs diagnosis, selects runbook. AWX executes Ansible or triggers Terraform Plan → Apply pipeline with human-in-loop gate for critical changes.

🔄 End-to-End Traffic Flow
Scenario: Trader at NYC branch accessing the EKS-hosted trading platform Source: 192.168.10.45 (branch VLAN 10 — Trading) → Destination: trading-app.internal.globalfinserv.com
CLIENT DNS Resolution
Trader's workstation queries branch DNS server → forwards to SD-WAN edge DNS proxy → Route53 private hosted zone resolves trading-app.internal to ALB private IP 10.0.1.100. Response cached at branch resolver TTL 60s.
BRANCH SD-WAN Path Selection
SD-WAN engine classifies traffic as DSCP EF (trading app — App-ID match on port 443 + SNI trading-app.internal). Policy: steer to MPLS tunnel (link quality: 4ms RTT, 0% loss). IPsec ESP packet encapsulated, forwarded to regional hub POP.
IPsec Tunnel Traversal
Packet exits branch as: Outer IP: 203.0.113.10 (branch WAN) → 198.51.100.50 (hub IPsec endpoint). Inner payload: 192.168.10.45:54321 → 10.0.1.100:443 (TLS). IKEv2 SA rekeyed every 8 hours. Tunnel MTU 1400 (accounting for ESP overhead).
NGFW Security Inspection
Hub decapsulates IPsec. Packet hits NGFW PA-5450. Zone check: branch-trust → cloud-app. Policy rule RULE-42 matches (src-zone, dst-zone, app=ssl, user=trader-group, dst=trading-platform-address-object). SSL Forward Proxy decrypts TLS 1.3 → scans payload → Threat Prevention profile checks for exploits. Re-encrypts, forwards.
VPC Cloud Ingress
Packet enters VPC via Transit Gateway attachment. Route table: 192.168.0.0/16 (branch supernet) → NGFW ENI. After inspection, 10.0.1.0/24 (public subnet) receives packet at ALB. ALB terminates TLS (ACM cert), performs health-checked routing to EKS NodePort service.
KUBERNETES Pod Routing
ALB routes to NodePort 30443 on worker node. kube-proxy NATs to ClusterIP 172.20.10.50:8443 (trading-app service). Calico enforces NetworkPolicy: only pods in namespace trading-app with label role: api may receive ingress on port 8443. Istio sidecar validates mTLS SPIFFE identity. Pod handles request.
LOG Telemetry Generation
At every hop: SD-WAN generates NetFlow/IPFIX → Splunk HF. NGFW generates traffic log (CEF format) → Splunk HF. ALB access log → S3 → Splunk S3 input. K8s audit log → Splunk Connect for K8s (Helm chart). All indexed in Splunk within 15 seconds of event.

🌍 SD-WAN Logic
Path Selection Policy (vendor-neutral)
# SD-WAN Application-Aware Routing Policy
# Applied per-branch at vCPE edge device

policy: TRADING-APP-SLA
  match:
    application: trading-platform     # App-ID via DPI
    destination: 10.0.0.0/8
    dscp:        EF (46)               # Expedited Forwarding
  preferred_path:  MPLS
  sla_thresholds:
    latency_ms:    50
    jitter_ms:     10
    packet_loss:   0.1%
  failover_order:  [MPLS, DIA, LTE]
  probe_interval:  300ms              # BFD echo packets
  switchover_time: < 1 second

policy: BULK-DATA-COST-OPT
  match:
    application:  [backup, file-transfer]
    dscp:         CS1 (low priority)
  preferred_path:  DIA
  failover_order:  [DIA, MPLS]

policy: VOICE-SLA
  match:
    application:  [sip, rtp, webex]
  preferred_path:  MPLS
  sla_thresholds:
    latency_ms:    100
    jitter_ms:     30
    mos_min:       3.6
Failover Behavior
  • Detection: BFD sessions every 300ms on all tunnels. Link state tracked independently per-path. No dependency on routing protocol convergence.
  • Switchover: When MPLS path SLA threshold exceeded for 2 consecutive probes, traffic instantly re-steered to DIA. ECMP load sharing available across equal-quality paths.
  • Re-convergence: MPLS path reinstated when it passes SLA for 10 consecutive probes (~3 seconds). Hysteresis prevents flapping.
  • LTE fallback: LTE modem pre-provisioned in warm-standby. Zero-touch provisioning via SD-WAN controller. Activates if both MPLS and DIA fail. Bandwidth limit: 50Mbps. Only critical app traffic (trading, voice) routed over LTE.
  • Telemetry: All path switches generate Syslog → Splunk with tunnel ID, previous path, new path, SLA metrics at time of failure.

🔄 Automation Workflow
EVENT-DRIVEN REMEDIATION PIPELINE

[1] SPLUNK ES → Correlation rule fires (threshold exceeded) → Notable Event created → REST webhook POST to Kafka topic: secops.alerts.critical
│ (Kafka consumer — FastAPI AI service)
[2] AI AGENT → Consumes alert JSON → Fetches last 15min of related telemetry from Splunk API → Builds context window (network state + app state + incident history) → Calls Claude API with structured prompt → Receives RemediationPlan JSON
│ (decision gate)
[3] APPROVAL GATE Severity CRITICAL + confidence > 0.85 + no_approval_required = AUTO-EXECUTE Otherwise → PagerDuty alert + Slack interactive approval button (15min timeout) Timeout with no response = execute rank-1 recommendation automatically
│
[4] EXECUTION → AWX (Ansible Tower) API → Launch job template with AI-provided vars OR → Terraform Cloud API → Queue Plan → Auto-Apply (for infra changes) OR → kubectl apply via in-cluster service account (for K8s changes)
│
[5] VALIDATION → Post-remediation check (60s delay) Re-fetch metrics from Splunk / Prometheus → Compare against SLA thresholds PASS → Close incident, log to Splunk (action=remediated, outcome=success) FAIL → Escalate to human, lock out further auto-execution for this incident
Human-in-the-loop gates: Changes to firewall zone rules, Terraform destroy operations, and credential rotation ALWAYS require human approval via Slack, regardless of AI confidence score. Auto-execution is scoped to reversible actions only.

🔍 Architecture Analysis
Why This Architecture Works
  • Defense in depth: No single security control is a single point of failure. NGFW at edge + K8s NetworkPolicy + Istio mTLS + WAF means 4 independent layers must be bypassed for a breach.
  • Closed-loop automation: Detect → Diagnose → Remediate → Validate happens entirely without human intervention for 80% of operational events, eliminating alert fatigue.
  • IaC as truth: All state lives in Git. Any drift is detected and corrected. No snowflake infrastructure.
  • Application-aware transport: SD-WAN with App-ID means the network understands what matters to the business and protects it accordingly.
Trade-offs & Honest Limitations
  • Complexity: This architecture has 7 integrated layers. Requires strong operational maturity. A team of 3–5 skilled engineers to maintain.
  • AI confidence risk: Auto-execution on AI recommendations introduces blast radius if the model hallucinates. Mitigated by: confidence thresholds, scoping auto-execution to reversible actions, and post-remediation validation.
  • Splunk cost: At 40GB/day ingestion, Splunk Enterprise licensing is ~$200–400K/year. Alternatives: OpenSearch + Grafana for cost reduction at the expense of ES maturity.
  • SD-WAN vendor lock-in: Cisco or Fortinet SD-WAN creates dependency. Vendor-neutral abstraction layer (Terraform + Ansible) mitigates partially.
Scalability Considerations
  • Branch scale: SD-WAN controller supports 1,000+ edges. Zero-touch provisioning via Ansible + SD-WAN onboarding API.
  • K8s scale: Karpenter node autoprovisioning handles bursty trading load. KEDA scales pods on custom metrics (order queue depth).
  • Splunk scale: Distributed deployment with indexer clustering. SmartStore offloads cold data to S3 at ~$23/TB/month vs hot storage.
  • Multi-region: AWS Global Accelerator can be added to route NYC→eu-west-1 for EMEA branches, reducing latency by 40ms on transatlantic paths.
Cost Considerations (Annual)
  • AWS (EKS + VPC + ALB + TGW): ~$180K/year for prod scale
  • Splunk Enterprise Security: ~$250K/year (40GB/day)
  • Palo Alto NGFW (PA-5450 x2): ~$120K/year (HW + subscriptions)
  • SD-WAN (42 branches): ~$80K/year (controller SaaS + vCPE licensing)
  • MPLS circuits (42 branches): ~$420K/year
  • Total TCO: ~$1.1M/year. Offset by: 95% downtime reduction on a platform processing $50M/day in trades.
